# Web Shell

## Architecture

CK.Operator generates a minimal `index.html` that bootstraps the standard web shell from CK.Lib.Js on the CK volume. The web shell reads the project's kernel declarations to build the UI dynamically.

```
index.html (generated by CK.Operator, ~30 lines)
  +-- loads /cklib/ck-client.js (NATS WSS client)
  +-- loads /cklib/ck-page.js (page lifecycle)
  +-- inline config: { kernels, auth, nats }
       +-- ck-page.js builds:
           |-- Header (project name, URN, connection status, auth)
           |-- Action sidebar (from kernel spec.actions)
           |-- Parameter form (per action)
           +-- Results panel (NATS messages, chat/body/envelope modes)
```

## UI Layout

Three-panel layout:

| Panel | Width | Content |
|-------|-------|---------|
| Action sidebar | 160px | Kernel actions (common + unique), NATS topics, history |
| Parameter form | 280px | Per-action input fields, JSON preview, send button |
| Results | flex | NATS messages in chat, body, or envelope view mode |

## Auth Integration

- Login button in header triggers Keycloak password grant via `ck-client.js`
- JWT stored in memory, auto-refresh 30s before expiry
- Actions with `access: auth` show lock icon, disabled when anonymous
- Bearer token attached to all NATS message headers

## NATS Subscriptions

The web shell subscribes to:
- `result.{kernel}` -- action results
- `event.{kernel}` -- lifecycle events
- `stream.{kernel}` -- per-token Claude streaming events (v3.6)

For each kernel in the project.

## kernel.create via Web UI

With the web shell and auth in place, authenticated users can dispatch kernel lifecycle actions directly from the browser:

1. User logs in (JWT obtained)
2. Selects `kernel.create` action in sidebar
3. Fills parameters: `name`, `type`, `template`
4. Sends message to `input.CK.Operator` via NATS WSS
5. CK.Operator creates kernel directory (9 awakening files)
6. CK.Operator creates ConceptKernel CR
7. New kernel appears in `kubectl get ck` and in the web sidebar on next status refresh

## Conformance

- CK.Operator MUST generate an index.html that loads CK.Lib.Js from the CK volume
- The web shell MUST build its action sidebar from the kernel's declared actions
- The web shell MUST support anonymous and authenticated modes
- Auth config (issuer, realm, client_id) MUST be injected by the operator, not hardcoded
- `kernel.create` and `kernel.spawn` MUST require `access: auth`
- The web shell MUST be capable of dispatching any declared action to any kernel in the project
